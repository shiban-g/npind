# Numba JITコンパイル時のコーディング規約

このドキュメントは、`numba.njit` でコンパイルする関数を実装する際に遭遇した問題とその解決策をまとめたものです。

## 問題: JITコンパイル時のタプルのサイズエラー

Numba の JIT コンパイラは、コンパイル時に関数の引数や変数の方を静的に決定しようとします。タプル (`tuple`) を扱う際、その**長さがコンパイル時に不定である**と、型推論に失敗し `TypingError` が発生します。

### 失敗するパターン

以下のようなコードは、`axis` や `my_list` の長さが実行時まで決まらないため、コンパイルに失敗します。

1.  **変数によるタプルのスライス**

    ```python
    # axis は実行時に決まる変数
    # スライス後のタプルの長さが不定になるためエラー
    my_tuple = (1, 2, 3, 4)
    sliced_tuple = my_tuple[:axis]
    ```

2.  **可変長のリストからタプルへの変換**

    ```python
    # my_list の要素数が実行時まで決まらない場合、
    # 生成されるタプルの長さも不定になるためエラー
    my_list = [i for i in range(n)] # n は実行時に決まる
    my_tuple = tuple(my_list)
    ```
3.  **ループ内でのタプルのサイズ変更**

    ```python
    # res がループのたびにサイズが変わるためエラー
    res = ()
    for i in range(n):
        res = res + (i,)
    ```


## 解決策: タプルの長さを変えない操作

この問題を回避するには、JIT コンパイルされる関数内で、**タプルの長さを常に一定に保つ**ようにコードを記述する必要があります。

### 成功するパターン: タプルの回転

タプルの要素を操作する有効な方法は「回転」です。`src/npind/take_along_axis.py` の `_take_along_axis_kernel` 関数では、この手法を用いてインデックス用のタプルを構築しています。

```python
# 1. 対象の要素が先頭に来るようにタプルを回転させる
temp_idx = out_idx
for _ in range(axis):
    temp_idx = temp_idx[1:] + temp_idx[:1]

# 2. 先頭の要素を新しい値で置き換える
new_idx_rotated = (new_value,) + temp_idx[1:]

# 3. 元の順序に戻すために逆回転させる
src_idx = new_idx_rotated
for _ in range(axis):
    src_idx = src_idx[-1:] + src_idx[:-1]
```

この方法では、ループ内で生成されるすべての中間タプル (`temp_idx`, `new_idx_rotated`, `src_idx`) の長さが元の `out_idx` と同じになります。これにより、Numba はコンパイル時にタプルの長さを静的に解決でき、型推論エラーを回避できます。

このテクニックは、`get_result_shape` 関数でも同様に利用されています。
